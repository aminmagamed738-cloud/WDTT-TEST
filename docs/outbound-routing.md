# Выходной IP и прокси

Модуль на вкладке **«Деплой»** управляет выходом только для WDTT-пользователей. Он не должен менять глобальный `default route` основного сервера: SSH, системные обновления, Docker, Telegram-бот и сам транспорт WDTT продолжают идти обычным маршрутом сервера.

## Общий принцип

WDTT-сервер поднимает клиентскую сеть `10.66.66.0/24` на интерфейсе `wdtt0`. Внешние режимы применяются только к этой подсети:

- внешний SOCKS5/HTTP использует `redsocks` и цепочку `WDTT_PROXY_OUT` в `iptables` только для `-i wdtt0`;
- WireGuard exit использует интерфейс `wg-wdtt-exit`, отдельную routing table `100` и `ip rule from 10.66.66.0/24 table 100`;
- возврат в direct очищает только правила WDTT Plus и не трогает остальные правила сервера.

Одновременно активен один внешний выход: `direct`, `external_proxy`, `wireguard_vps` или `imported_wg`.

## 1. Прокси на этом VPS

Кнопка **«Прокси на этом VPS»** ставит `3proxy`, создаёт `wdtt-3proxy.service` и включает SOCKS5/HTTP-прокси с логином и паролем. Это прокси для администратора или внешних клиентов, оно не меняет маршрут WDTT-пользователей само по себе.

Важно:

- open proxy не создаётся, авторизация обязательна;
- SOCKS5-порт задаётся в окне, HTTP-порт создаётся как `SOCKS5 + 1`;
- пароль не пишется в обычный Android-лог, в результате показывается URI с замаскированным паролем.

Проверка выполняется через:

```bash
curl --socks5-hostname LOGIN:PASSWORD@127.0.0.1:PORT https://api.ipify.org
```

## 2. Внешний SOCKS5/HTTP-прокси

Кнопка **«Внешний SOCKS5/HTTP»** ставит `redsocks`, поднимает редиректор на `0.0.0.0:12345` и перенаправляет TCP-трафик WDTT-пользователей через внешний прокси.

Ограничения:

- режим рассчитан в основном на TCP;
- UDP может не поддерживаться;
- качество, страна и репутация IP зависят от внешнего прокси;
- private/local ranges исключены из перенаправления.

Если ICMP-пинги проходят, а сайты не открываются, проверьте диагностику: для трафика из `PREROUTING` редиректор не должен слушать только `127.0.0.1:12345`.

Этот режим не меняет default route VPS.

## 3. Зарубежный VPS через WireGuard

Кнопка **«WG exit VPS»** подключается по SSH к foreign VPS, ставит WireGuard, создаёт туннель `wg-wdtt-exit` между текущим WDTT-сервером и foreign VPS, включает NAT на foreign VPS и policy routing на текущем сервере.

Это рекомендуемый полноценный режим для смены выходного IP WDTT-пользователей, потому что WireGuard переносит не только TCP.

## 4. Импорт WireGuard config

Кнопка **«Импорт WG config»** принимает готовый `.conf` файлом или текстом. Конфиг проверяется на обязательные поля:

- `[Interface] PrivateKey`
- `[Interface] Address`
- `[Peer] PublicKey`
- `[Peer] Endpoint`
- `[Peer] AllowedIPs`

`PreUp`, `PostUp`, `PreDown`, `PostDown` запрещены при импорте, чтобы чужой конфиг не мог поменять маршруты всего VPS. WDTT Plus добавляет `Table = off`, поднимает `wg-wdtt-exit` и маршрутизирует через него только WDTT-подсеть.

`AllowedIPs = 0.0.0.0/0` нормально для exit-node: благодаря `Table = off` и отдельному `ip rule` это не становится default route всего сервера.

## Диагностика и rollback

Кнопки **«Статус»** и **«Диагностика»** показывают:

- текущий outbound mode;
- WDTT client subnet;
- внешний IP самого сервера;
- состояние `3proxy`, `redsocks`, `wg-wdtt-exit`;
- `ip rule`;
- routing table `100`;
- WDTT-цепочки `iptables`.

Кнопка **«Вернуть direct»** останавливает `redsocks`, опускает `wg-wdtt-exit`, чистит policy routing и возвращает прямой выход через текущий VPS.
